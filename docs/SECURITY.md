# Security

## Authentication

Elowen uses **Bearer token** authentication. Every route except `GET /health`
and `POST /auth/login` requires a valid token.

```
Authorization: Bearer <token>
```

### Token types

| Type | Generated by | Usage |
|------|-------------|-------|
| `full` | Login | Interactive sessions (web UI, CLI) — full CRUD |
| `agent` | Daemon | Spawned agents — restricted verb allow-list |
| `advisor` | Daemon | Per-user assistant — mapped to `full` rights |

### Token storage

- **Web UI**: httpOnly session cookie via the same-origin BFF proxy. The token
  never reaches browser JavaScript.
- **CLI**: `~/.config/elowen/token` with `0600` permissions.
- **Agent**: Environment variable — scoped to the agent's process.

### Password policy

- Minimum 8 characters
- Hashed with scrypt + random 16-byte salt
- Changed via `POST /auth/me/password` (requires current password)

## Access control

### Roles

| Role | Description |
|------|-------------|
| **Admin** | Full access — all projects, users, settings |
| **Member** | Restricted to assigned projects and allowed executors |

### Multi-tenancy gates

1. **Global gate** — non-admin must be assigned to the daemon's home project
2. **Per-project gate** — users only see/operate assigned projects
3. **Per-user exec allowlist** — restricts which executors a non-admin may use

### Agent scope confinement

Agent-scoped tokens are confined to:
- Their own working project (cross-tenancy forbidden)
- A narrow verb allow-list (close tasks, submit plans, read sessions)
- Their own tasks only

## API security

### Rate limiting

- Login: 10 attempts per 5 minutes per IP (prefers `x-real-ip` header)
- In-memory counter, resets on daemon restart

### CORS

Enabled for all origins by default. Restrict in code for production.

### Headers

The daemon prefers `x-real-ip` over `x-forwarded-for` for rate limiting
and audit logging.

## Push notification security

- VAPID keys auto-generated on first boot, persisted in config store
- Private key never leaves the daemon
- Dead push subscriptions (404/410) pruned automatically
- Subscription endpoints are per-user, scoped to authenticated session

## Plugin security

### Path guard

All file-system plugins use `ctx.assertPathAllowed(path)` which resolves
symlinks and checks the path is within the user's allowed project roots.

### SSRF guard

The web plugin checks DNS resolution against private IP ranges before
fetching URLs. Rate-limited to prevent abuse.

### Owner-only tools

Certain tools (`DiscordApi`, `Bash` in shared channels) are gated
to the true owner (`identity.owner === true`), not merely admin roles.

### Secrets handling

Plugin config marked as `secret` in the schema is write-only — the daemon
returns `true`/`false` for whether a key has been set, never the value.

## Config security

### API keys

All API keys (brain providers, relay, GitHub, plugins) are write-only.
The config endpoint exposes `apiKeySet: boolean` for each key, never the
value.

### Allowed executors

`config.allowedExecs` is a global allow-list. Unknown executors are rejected
at the API boundary before any processing.

## Database

SQLite with WAL mode. Database file permissions should be restricted to the
daemon's user (`chmod 600` recommended for `~/.config/elowen/elowen.db`).
